# 余额商城与运营工具设计

## 目标

把现有“选择套餐后提交管理员审核”改为“余额即时付款”，同时补齐一次性充值卡、管理员调账、用户备注与筛选、多端订阅链接，以及套餐/面板弹窗编辑体验。

## 产品决策

- 金额以人民币展示，数据库统一使用整数分 `*_cents`，避免浮点误差。
- 套餐购买立即生效：事务内校验余额、扣款、替换套餐、刷新额度与有效期，并记录余额流水；事务完成后执行 X-UI 开通。
- 每次购买开启新的用量周期，旧周期用量清零。购买失败、余额不足或套餐停售时不扣款。
- 充值卡由管理员指定金额和数量批量生成。完整卡号只在生成结果中展示一次；数据库保存哈希和尾号。兑换成功后永久作废，并记录兑换用户与时间。
- 管理员可以给指定用户增加或扣减余额，必须填写原因；余额不得变成负数。
- 用户备注由备注文本和重点标记组成，管理员可在列表直接编辑。搜索匹配邮箱和备注；筛选支持状态、账号类型、重点用户。
- 多端订阅提供 Clash/Clash Meta、通用 Base64 分享链接和 sing-box JSON 三类地址，并在界面标明 macOS、Android、iOS 的常见客户端用途。
- 套餐和 X-UI 面板的新增、编辑统一使用原生 `dialog` 弹窗；列表标题旁放置“新增”按钮，配置页首屏不再堆叠两个大表单。
- 登录后商城首页不展示账号邮箱；账户身份只在“我的订阅”或后台操作区中出现。

## 数据模型

### 现有表扩展

- `plans.price_cents integer not null default 0`
- `users.balance_cents integer not null default 0`
- `users.admin_note text not null default ''`
- `users.is_priority integer not null default 0`

### 新表

`balance_transactions` 保存所有余额变化：用户、变化金额、变化后余额、类型、说明、关联充值卡和创建时间。

`recharge_cards` 保存卡密哈希、显示尾号、面额、状态、创建/兑换用户和时间。卡密使用高熵随机值，兑换查询使用 SHA-256 哈希。

## 后端接口

- `POST /api/purchases`：登录用户余额购买套餐。
- `POST /api/recharge`：登录用户兑换充值卡。
- `GET /api/balance/transactions`：登录用户读取余额流水。
- `GET /api/admin/recharge-cards`：管理员读取充值卡记录。
- `POST /api/admin/recharge-cards`：管理员批量生成充值卡并返回本次完整卡号。
- `POST /api/admin/users/balance`：管理员调账。
- `POST /api/admin/users/note`：管理员更新备注与重点标记。
- `/sub/clash/{token}`：Clash YAML。
- `/sub/base64/{token}`：V2Ray/Shadowrocket/Hiddify 等支持的分享链接集合。
- `/sub/singbox/{token}`：sing-box JSON。

## 界面与交互

视觉命题：延续现有浅色运营后台和蓝色单一强调色，以更紧凑的列表、抽屉式细节和轻量弹窗减少滚动。

内容计划：商城套餐价格与余额购买；账户页余额、充值与多端订阅；后台用户工具栏与可折叠详情；配置页列表加弹窗；独立充值卡运营区。

交互命题：弹窗使用短促淡入/缩放；用户详情使用折叠高度过渡；购买、充值、生成卡密使用现有计时加载动画并锁定提交按钮。

桌面端用户表格保持高密度，点击行或“详情”展开备注、余额调账和生命周期操作。移动端使用同一筛选器与折叠卡片。搜索和筛选即时生效并显示结果数量。

## 一致性与错误处理

- 所有余额与充值卡写操作使用 `begin immediate` 事务，保证并发兑换和购买只能成功一次。
- API 不返回卡密哈希、密码或内部凭据。
- 余额不足、卡密无效/已使用、停售套餐、管理员扣款导致负余额均返回清晰的 400 错误。
- 余额已扣但远端开通部分失败时保留购买结果，在账户与后台展示开通失败详情并允许重试，避免重复扣款。

## 测试范围

- 数据库迁移兼容旧库；余额购买原子性、用量重置、充值卡一次性、管理员调账和备注。
- API 鉴权、敏感字段脱敏、错误分支与购买后开通。
- Clash、Base64、sing-box 三种订阅的有效响应和停用状态。
- 前端弹窗、隐藏首页账号、多端链接、充值卡、搜索筛选、折叠和备注控件存在并绑定接口。
